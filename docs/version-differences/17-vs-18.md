# React 17 vs React 18 - Key Differences

This guide explains the major differences between React 17 and React 18, helping you understand when and how to upgrade.

## 🎯 TL;DR - What Changed?

React 18 introduces **concurrent rendering** - the ability to prepare multiple versions of the UI at the same time, allowing React to keep your app responsive even during expensive updates.

## Major Changes

### 1. New Root API - `createRoot`

**React 17:**
```jsx
import ReactDOM from 'react-dom';

ReactDOM.render(<App />, document.getElementById('root'));
```

**React 18:**
```jsx
import { createRoot } from 'react-dom/client';

const root = createRoot(document.getElementById('root'));
root.render(<App />);
```

**Why it matters:**
- The old `ReactDOM.render()` still works but runs in "legacy mode"
- `createRoot` enables all concurrent features
- You MUST use `createRoot` to get React 18 benefits

**Migration:**
```bash
# Find and replace in main.jsx/index.jsx
- import ReactDOM from 'react-dom';
+ import { createRoot } from 'react-dom/client';

- ReactDOM.render(<App />, rootElement);
+ const root = createRoot(rootElement);
+ root.render(<App />);
```

### 2. Automatic Batching

**React 17 Behavior:**
```jsx
// ✅ BATCHED (1 re-render)
function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
}

// ❌ NOT BATCHED (2 re-renders)
function handleClick() {
  setTimeout(() => {
    setCount(c => c + 1);
    setFlag(f => !f);
  }, 0);
}

// ❌ NOT BATCHED (2 re-renders)
function handleClick() {
  fetch('/api').then(() => {
    setCount(c => c + 1);
    setFlag(f => !f);
  });
}
```

**React 18 Behavior:**
```jsx
// ✅ ALL BATCHED (1 re-render each)
function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
}

setTimeout(() => {
  setCount(c => c + 1);  // ✅ Batched!
  setFlag(f => !f);
}, 0);

fetch('/api').then(() => {
  setCount(c => c + 1);  // ✅ Batched!
  setFlag(f => !f);
});
```

**Why it matters:**
- Fewer re-renders = better performance
- More consistent behavior
- No special logic needed for async updates

**Opting out when needed:**
```jsx
import { flushSync } from 'react-dom';

flushSync(() => {
  setCount(c => c + 1);  // Renders immediately
});
flushSync(() => {
  setFlag(f => !f);  // Renders immediately (separate)
});
```

### 3. New Hooks

#### `useTransition` - Non-urgent Updates

**Problem in React 17:**
```jsx
// Both updates render together, blocking the UI
function handleChange(e) {
  setInput(e.target.value);      // Urgent (user is typing)
  setFilteredList(filter(list)); // Expensive, but not urgent
}
```

**Solution in React 18:**
```jsx
const [isPending, startTransition] = useTransition();

function handleChange(e) {
  setInput(e.target.value);  // Urgent - renders immediately

  startTransition(() => {
    setFilteredList(filter(list));  // Non-urgent - can be interrupted
  });
}

// Show loading state while transition is pending
{isPending && <Spinner />}
```

**Benefits:**
- UI stays responsive during expensive operations
- React can pause/resume low-priority work
- User interactions never feel sluggish

---

#### `useDeferredValue` - Debouncing Without Timers

**React 17 Pattern:**
```jsx
const [input, setInput] = useState('');
const [debouncedInput, setDebouncedInput] = useState('');

useEffect(() => {
  const timer = setTimeout(() => {
    setDebouncedInput(input);
  }, 300);
  return () => clearTimeout(timer);
}, [input]);

// Use debouncedInput for expensive operations
const filteredList = expensiveFilter(list, debouncedInput);
```

**React 18 Alternative:**
```jsx
const [input, setInput] = useState('');
const deferredInput = useDeferredValue(input);

// Expensive operation uses deferred value
const filteredList = expensiveFilter(list, deferredInput);

// Optional: show "stale" indicator
const isStale = deferredInput !== input;
```

**When to use each:**
- `useTransition`: You control the state update
- `useDeferredValue`: You receive a prop/value and want to defer it

---

#### `useId` - Stable IDs for Accessibility

**Problem in React 17:**
```jsx
// ❌ Not SSR-safe - ID differs between server and client
const id = Math.random().toString();

// ❌ Not unique across multiple instances
const id = 'form-input';
```

**Solution in React 18:**
```jsx
const id = useId();

return (
  <>
    <label htmlFor={id}>Email</label>
    <input id={id} type="email" />
  </>
);
```

**Benefits:**
- Stable across server and client renders
- Automatically unique per component instance
- Perfect for accessibility (ARIA IDs)

---

#### `useSyncExternalStore` - Subscribe to External State

**For library authors:**
```jsx
function useWindowWidth() {
  return useSyncExternalStore(
    (callback) => {
      window.addEventListener('resize', callback);
      return () => window.removeEventListener('resize', callback);
    },
    () => window.innerWidth,
    () => 0 // Server snapshot
  );
}
```

**Why it exists:**
- Ensures external stores stay consistent during concurrent rendering
- Prevents "tearing" (UI showing mixed old/new state)
- Most developers won't use this directly (it's for libraries)

---

### 4. Strict Mode Changes

**React 17:**
- Warnings about deprecated APIs
- Detects side effects in render

**React 18 (Development Only):**
- **Double-invokes effects** to catch missing cleanup functions
- Mount → Unmount → Remount components

```jsx
useEffect(() => {
  console.log('mounted');

  return () => {
    console.log('cleanup');
  };
}, []);

// React 17 output:
// mounted

// React 18 output (dev mode):
// mounted
// cleanup
// mounted
```

**Why?**
- Catches bugs where you forget cleanup
- Prepares for future features (Fast Refresh improvements)
- Only in development mode

---

### 5. Suspense Improvements

**React 17:**
- Suspense works for `React.lazy()` code splitting
- **Not** ready for data fetching

**React 18:**
- Enhanced Suspense for data fetching (with frameworks)
- Better streaming SSR support
- Multiple Suspense boundaries coordinate better

## Migration Checklist

### Step 1: Update Dependencies

```json
{
  "dependencies": {
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  }
}
```

### Step 2: Update Root API

```jsx
// main.jsx or index.jsx
import { createRoot } from 'react-dom/client';

const root = createRoot(document.getElementById('root'));
root.render(<App />);
```

### Step 3: Test in Development

```bash
npm run dev
```

Watch for console warnings about:
- Missing effect cleanup functions
- Deprecated APIs
- State updates after unmount

### Step 4: Fix Effect Cleanup

```jsx
// ❌ Missing cleanup
useEffect(() => {
  const id = setInterval(() => { /* ... */ }, 1000);
}, []);

// ✅ Proper cleanup
useEffect(() => {
  const id = setInterval(() => { /* ... */ }, 1000);
  return () => clearInterval(id);
}, []);
```

### Step 5: Remove Manual Batching

```jsx
// ❌ React 17 pattern - no longer needed
import { unstable_batchedUpdates } from 'react-dom';

unstable_batchedUpdates(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
});

// ✅ React 18 - automatic!
setCount(c => c + 1);
setFlag(f => !f);
```

### Step 6: Opt Into Concurrent Features

Add `useTransition` or `useDeferredValue` where appropriate:

```jsx
// Before: blocking update
setSearchResults(expensiveFilter(items, query));

// After: non-blocking update
startTransition(() => {
  setSearchResults(expensiveFilter(items, query));
});
```

## Breaking Changes

### 1. Consistent `useEffect` Timing

Effects now consistently run **after** layout and paint, even in legacy mode.

### 2. Stricter Hydration

React 18 is stricter about server/client mismatches. Fix warnings or they'll become errors.

### 3. Automatic Batching

If your code relied on multiple renders, it might break. Use `flushSync` if needed:

```jsx
import { flushSync } from 'react-dom';

flushSync(() => setCount(1));
// DOM updated immediately
ref.current.scrollIntoView();
```

### 4. Suspense Behavior

Suspense no longer requires a fallback in all cases. This might affect your layouts.

## Performance Comparison

| Scenario | React 17 | React 18 |
|----------|----------|----------|
| Event handler updates | 1 render | 1 render (same) |
| setTimeout updates | 2+ renders | 1 render ✅ |
| Promise updates | 2+ renders | 1 render ✅ |
| With useTransition | N/A | Non-blocking ✅ |
| Large list rendering | Blocks UI | Interruptible ✅ |

---

## When to Upgrade?

**Upgrade if:**
- ✅ You want better performance
- ✅ You have expensive re-renders
- ✅ Your UI feels sluggish during updates
- ✅ You're starting a new project

**Wait if:**
- ⏸️ Your app is stable and performant
- ⏸️ You use old dependencies that don't support React 18
- ⏸️ You don't have time to test thoroughly
